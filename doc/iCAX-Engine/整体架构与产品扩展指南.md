# iCAX 整体架构与产品扩展指南

## 1. 文档定位

本文是 iCAX 当前阶段的总入口文档，用来说明：

- C++ backend 与 H5 frontend 的整体分工。
- Application、Product、Project 之间的运行关系。
- Mailbox、PDO、Resource、Database、Behaviour、Service、CommandHandler 在系统中的位置。
- 如何扩展实现一款自己的产品。

更细的模块规格仍以各模块目录下的规格文档和方案文档为准。本文只负责把全局视角串起来。

## 2. 总体定位

iCAX 是一个“C++ 后台 + H5 前台”的进程内应用框架。

```text
Process
  C++ Backend
    ApplicationHost
      ProductRuntime*
        Project*
          Repository
          ResourceLibrary
          Universe
          MailChannel
          PDO

  H5 Frontend
    FrontendHost
      ApplicationWorkspace
        ProductWorkspace*
          ProjectWorkspace*
```

当前基本原则：

- backend 负责真实数据、资源、行为、命令、项目生命周期和持久化。
- frontend 负责界面、交互、轻量状态缓存和用户操作编排。
- frontend 不直接访问 C++ 对象，不直接持有 Repository 或 ResourcePool。
- frontend 与 backend 通过宿主 bridge 连接。
- 普通请求响应走 Mailbox。
- 高频状态走 PDO。
- 大资源不塞进 Mail payload，而是通过 Resource 机制或宿主资源 URL 暴露。

## 3. 目录结构

```text
src/
  icax/                  C++ 后台平台
    foundation/          基础能力，不依赖 framework
    framework/           后台核心框架
    frontend/            WebPageHost + WebPageShell + FrontendSDK
  tests/                 测试

  apps/                  具体产品
    {product}/
    product.manifest.json
    backend/
    webpage/
    protocol/
```

`icax-engine` 和 `iCAX-UI` 不是某个具体产品，它们分别是后台平台和公共前端框架。具体产品放在 `src/apps/{product}`，产品自己的 backend、webpage 和 protocol 都归属该产品目录。

## 4. Backend 层级

### 4.1 ApplicationHost

`ApplicationHost` 是 backend 的应用级入口，也是 frontend 启动后最先连接的稳定对象。

职责：

- 加载 `ApplicationContext` 和应用级配置。
- 创建应用级 `CServiceProvider`。
- 创建应用级 `IMailChannelService`。
- 维护支持的产品定义列表。
- 创建 application mail channel。
- 处理应用级命令，例如 `App.GetState`、`App.StartProduct`、`App.OpenProjectFile`。
- 启动和停止 `ProductRuntime`。

ApplicationHost 不直接维护项目数据，也不直接打开 Project。项目归属 ProductRuntime。

### 4.2 ProductRuntime

`ProductRuntime` 是产品级运行时。

职责：

- 持有产品静态定义 `CProductDefinition`。
- 加载产品数据 `CProductData`，例如最近打开项目。
- 加载产品模块 DLL。
- 回放产品模块中的 ComponentMeta、Behaviour、Service、ResourceLoader 注册。
- 创建 product mail channel。
- 处理产品级命令，例如 `Product.GetState`、`Product.OpenProjectCatalog`。
- 管理 ProjectCatalog 和 ProjectRuntime。

ProductRuntime 本身不创建独立工作线程。当前由 ApplicationHost 的应用级工作线程轮询产品级 mailbox。

### 4.3 Project

`Project` 是数据和运行隔离边界。

一个 Project 至少拥有：

- `Repository`：EC 数据容器。
- `ResourceLibrary`：项目资源库。
- `Universe`：行为执行容器。
- `MailChannel`：项目级通信通道。
- 项目工作线程。

不同 Project 之间的数据、资源对象、Universe 实例和项目线程互相隔离。

### 4.4 Repository

`Repository` 是 EC 数据容器。

- Entity 表示对象身份。
- Component 表示实体的身份、能力或状态。
- ComponentMeta 描述字段的持久化、变更策略、派生字段等信息。
- Repository 负责实体组件管理、事件、变更日志、事务、撤销重做和快速保存日志。

Repository 不再用 Domain/World 做额外隔离。需要临时编辑、预览、导入时，应优先考虑轻量 Project。

### 4.5 ResourceLibrary

资源属于 Project。

```text
Project
  ResourceLibrary
    ResourcePool
```

资源 loader 的定义在应用级注册，资源实例和资源池按 Project 隔离。上层使用时应以资源类型和 source 路径作为主要身份，例如：

```cpp
auto model = project.Resources().Load<CModelResource>("D:/assets/robot.fbx");
```

资源持久化策略由 `CResourceInfo::Persistence` 表达：

- `RuntimeOnly`：运行期资源，不保存到工程。
- `Embedded`：保存工程时内嵌资源。
- `External`：保存工程时记录外部引用。

## 5. Frontend 层级

H5 前端由 `src/iCAX-UI` 提供公共框架：

```text
WebPageHost
  WebPageShell
    ApplicationWorkspace
      ProductWorkspace
        ProjectWorkspace
```

### 5.1 WebPageHost

WebPageHost 是原生宿主，负责：

- 创建 ECF/WebView 页面。
- 启动并持有 `ApplicationHost`。
- 注入 `window.icax`。
- 管理 `channelId -> frontend post office`。
- 将 H5 mail 投递到 backend。
- 将 backend response/event 推送到 H5。
- 管理 PDO shared memory 读写租约。

### 5.2 WebPageShell

WebPageShell 是唯一 H5 页面入口，负责：

- 获取 application mail id。
- 创建 `MailboxClient`。
- 拉取 `App.GetState`。
- 维护前端 UI 状态。
- 挂载通用工作台布局。
- 根据产品 manifest 的 `webpage.entry` 动态加载产品前端模块。

### 5.3 ApplicationWorkspace

ApplicationWorkspace 负责：

- 展示支持的产品列表。
- 启动产品。
- 处理双击项目文件打开。
- 根据 application 状态决定显示产品列表还是产品工作台。

双击文件时，frontend 只传项目路径：

```js
await appClient.openProjectFile("D:/projects/robot.robot");
```

产品识别只由 backend 根据项目文件 magic 完成，frontend 不传 productId。

### 5.4 ProductWorkspace

ProductWorkspace 负责：

- 展示产品首页。
- 展示最近项目。
- 打开 ProjectCatalog。
- 展示已打开项目列表。
- 通过 `WebPageShell` 动态加载产品前端入口 `webpage.entry`。

产品级 mailbox 由 `App.StartProduct` 或 `App.OpenProjectFile` 返回的 `productChannelId` 建立。

### 5.5 ProjectWorkspace

ProjectWorkspace 负责：

- 项目主编辑区。
- 属性面板。
- 资源面板。
- 日志、任务、诊断。
- 项目级命令入口，例如保存、撤销、重做、执行业务命令。

项目级 mailbox 由 Project 创建后返回的 `projectChannelId` 建立。

## 6. 通信模型

### 6.1 Bridge

宿主需要向 H5 注入 bridge。H5 只依赖抽象接口，不依赖具体宿主技术。

```ts
interface IcaxBridge {
  getApplicationChannelId(): Promise<string>;
  postMail(channelId: string, mail: IcaxMail): void | Promise<void>;
  subscribeMail(channelId: string, handler: (mail: IcaxMail) => void): () => void;
  subscribePDO?(pdoId: string, handler: (value: unknown) => void): () => void;
  openFileDialog?(options: OpenFileDialogOptions): Promise<string | null>;
  onProjectFileOpen?(handler: (path: string) => void): () => void;
}
```

宿主可以是 WebView2、Qt WebEngine、CEF、WPF WebView，或者调试浏览器。

### 6.2 Mailbox

Mailbox 用于低频命令、请求响应、事件和状态同步。

```text
ApplicationHost
  IMailChannelService
    applicationChannelId -> CMailChannel
    productChannelId     -> CMailChannel
    projectChannelId     -> CMailChannel
```

每个运行体只知道自己的 mail id。`IMailChannelService` 负责按 id 管理 channel。

### 6.3 Promise 语义

frontend 和 backend 是两个线程。frontend 调用 `postMail` 后，不能立即得到结果。

前端所有命令 API 都返回 `Promise`：

```js
const state = await appClient.getState();
const opened = await appClient.openProjectFile("D:/projects/sample.icax");
```

完成规则：

- request mail 带唯一 `id`。
- request mail 的 `originId` 为 `0` 或空。
- backend 处理完成后发送 response mail。
- response mail 的 `originId` 等于 request mail 的 `id`。
- `stamp == "Ok"` 时 Promise resolve `payload`。
- `stamp != "Ok"` 时 Promise reject。
- 超时未收到 response mail 时 Promise reject timeout。

### 6.4 PDO

PDO 用于高频数据，例如：

- 鼠标拖动过程中的连续状态。
- 进度、仿真过程状态。
- 视图矩阵、渲染状态、实时测量结果。

PDO 的核心特征是可覆盖、可丢弃、可按帧采样。它不替代 Mailbox，也不承载事务型命令。

### 6.5 Resource

大资源不要通过 Mail payload 传输。推荐方式：

- backend 用 ResourceLibrary 管理资源。
- frontend 只保存资源 id/source/type。
- 真正显示时通过宿主资源 URL 或 Resource API 获取。

## 7. 典型运行流程

### 7.1 普通启动

```text
Host process
  -> create ApplicationHost
  -> ApplicationHost.SetConfig(config)
  -> ApplicationHost.Start()

H5 bootstrap
  -> bridge.getApplicationChannelId()
  -> App.GetState
  -> render product list

User selects product
  -> App.StartProduct(productId)
  -> backend creates ProductRuntime
  -> backend loads product modules
  <- productChannelId + frontendEntry + recentProjects
  -> frontend renders ProductWorkspace
```

### 7.2 双击项目文件启动

```text
Host receives file path
  -> frontend onProjectFileOpen(path)
  -> App.OpenProjectFile(projectPath)

ApplicationHost
  -> read project file magic
  -> resolve product
  -> start ProductRuntime if needed
  -> ProductRuntime.OpenProjectCatalog()
  -> create main Project
  <- product + catalog + projectChannelId

Frontend
  -> activate ProductWorkspace
  -> activate ProjectWorkspace
```

项目文件归属产品只由 magic 判断。扩展名只用于文件选择器和展示。

### 7.3 打开项目后的项目命令

```text
ProjectWorkspace
  -> projectClient.execute(command, payload)
  -> bridge.postMail(projectChannelId, requestMail)

Project thread
  -> receive mail
  -> ProductRuntime dispatches command with project context
  -> handler reads Repository / ResourceLibrary / Universe / Services
  -> response mail

Frontend
  -> Promise resolve/reject
  -> update UI state
```

## 8. 如何扩展自己的产品

### 8.1 推荐目录

```text
src/apps/robot/
  README.md
  product.manifest.json
  backend/
    component/
      RobotComponent/
    behaviour/
      RobotBehaviour/
    service/
      RobotService/
    command/
      RobotCommand/
  webpage/
    entry.mjs
    pages/
    panels/
    components/
  protocol/
    mailbox/
    pdo/
```

其中：

- `component` 定义产品数据结构。
- `behaviour` 定义数据如何驱动行为。
- `service` 定义可复用能力。
- `command` 定义前端命令到 backend 用例的映射。
- `webpage` 定义产品 UI。
- `protocol` 记录产品级命令名、payload、PDO 数据结构。

### 8.2 定义产品

产品必须有全局唯一的 `ProductID` 和项目文件 magic。

```cpp
iCAX::Product::CProductDefinition robot;
robot.ProductID = "robot";
robot.ProductName = "Robot";
robot.ProductVersion = "1.0";
robot.FrontendEntry = "apps/robot/webpage/entry.mjs";
robot.DefaultProjectStartupComponent = "RobotStartupComponent";

robot.ProjectFile.Magic = "ICAX_ROBOT";
robot.ProjectFile.FormatVersion = "1.0";
robot.ProjectFile.FileExtensions.push_back(".robot");
robot.ProjectFile.MagicOffset = 0;
robot.ProjectFile.ProbeBytes = 256;

robot.Modules.ComponentModules.push_back("bin/RobotComponent.dll");
robot.Modules.BehaviourModules.push_back("bin/RobotBehaviour.dll");
robot.Modules.ServiceModules.push_back("bin/RobotService.dll");
robot.Modules.CommandModules.push_back("bin/RobotCommand.dll");

iCAX::ApplicationHost::ApplicationHostConfig config;
config.Descriptor.AppID = "icax";
config.Descriptor.AppName = "iCAX";
config.Products.push_back(robot);
```

注意：

- `Magic` 是必须项。
- `Magic` 在同一个 ApplicationHost 下必须唯一。
- `FileExtensions` 不能作为产品识别依据。
- `DefaultProjectStartupComponent` 是打开项目时默认启动组件。

### 8.3 定义 Component

Component 表达 Entity 的身份、能力或状态。不要为了持久化、撤销重做或性能去拆 Component。

```cpp
class CRobotJointComponent final : public iCAX::Database::CComponentBase
{
    DECLARE_ICAX_COMPONENT(CRobotJointComponent, iCAX::Database::CComponentBase)
    DECLARE_ICAX_COMPONENT_CREATOR(CRobotJointComponent)

    DECLARED_ICAX_FIELD(
        CRobotJointComponent,
        double,
        Angle,
        0.0,
        [](double a, double b) { return a == b; },
        [](double value) { return iCAX::Data::Variant(value); },
        [](const iCAX::Data::Variant& value) { return value.To<double>(); })
};
```

字段类型建议按语义选择：

- `DECLARED_ICAX_FIELD`：持久化字段，参与事件、版本、事务、撤销重做。
- `DECLARED_ICAX_OBSERVABLE_FIELD`：非持久化可观察字段，影响版本并发布事件。
- `DECLARED_ICAX_SILENT_FIELD`：非持久化静默字段，正常发布事件，订阅者根据 meta 决定是否忽略撤销重做等。
- `DECLARED_ICAX_RUNTIME_FIELD`：运行时裸字段，不进入 Property/Meta 系统。
- `DECLARED_ICAX_DERIVED_FIELD`：派生字段，只读，按需计算，由 Database 维护过期状态。

### 8.4 定义 Behaviour

Behaviour 是 Component 的 system 部分。Behaviour 定义在产品级注册，但执行时使用当前 Project 的 Universe 和 Repository。

```cpp
class CRobotKinematicsBehaviour final : public iCAX::Behaviour::CBehaviourBase
{
    AUTO_REGIST_BEHAVIOUR(CRobotKinematicsBehaviour)

public:
    std::string GetComponentClass() const override
    {
        return "CRobotJointComponent";
    }

protected:
    void OnUpdate(
        iCAX::Database::CComponentBase& component,
        const iCAX::Application::IApplicationContext& application,
        const iCAX::Product::IProductContext& product,
        iCAX::Project::IProjectContext& project,
        const double& deltaTime,
        const double& totalTime) override
    {
        auto& joint = static_cast<CRobotJointComponent&>(component);
        // 根据 joint 和 project.Database() 计算本帧行为。
    }
};
```

产品模块 DLL 被 ProductRuntime 加载后，`AUTO_REGIST_BEHAVIOUR` 会进入进程级注册目录，并在产品启动时按模块路径回放到当前 ProductRuntime 的产品级 BehaviourRegistry。注册目录是追加式的，不提供注销或插件热卸载能力；项目关闭只解除当前 Universe 内的 Behaviour 实例。

### 8.5 定义 Service

Service 是应用级共享能力。它适合表达跨产品、跨项目可复用的基础能力，例如日志、配置、文件系统、授权、算法库入口等。

```cpp
class IRobotSolverService : public iCAX::Services::IService
{
public:
    virtual ~IRobotSolverService() = default;
    virtual void Solve() = 0;
};

class CRobotSolverService final : public IRobotSolverService
{
    AUTO_REGIST_SERVICE(IRobotSolverService, CRobotSolverService)

public:
    void Solve() override
    {
        // 产品算法入口
    }
};
```

Service 当前是 Application 级共享，不是 Project 级数据容器。需要按项目隔离的数据应放入 Project、Repository 或 ResourceLibrary。

### 8.6 定义 ResourceLoader

资源 loader 负责把 source 加载为某种资源类型。

```cpp
class CRobotModelResource final
{
public:
    std::string Source;
};

class CRobotModelLoader final : public iCAX::Resource::IResourceLoader
{
public:
    bool CanLoad(const iCAX::Resource::CResourceLoadContext& context) const override
    {
        return context.Source.ends_with(".fbx");
    }

    iCAX::Resource::CResourceLoadResult Load(
        const iCAX::Resource::CResourceLoadContext& context) override
    {
        auto resource = std::make_shared<CRobotModelResource>();
        resource->Source = context.Source;

        iCAX::Resource::CResourceInfo info = context.Info;
        info.Source = context.Source;
        return iCAX::Resource::CResourceLoadResult::Succeeded(info, resource);
    }
};

ICAX_REGISTER_RESOURCE_LOADER(CRobotModelResource, CRobotModelLoader)
```

上层使用：

```cpp
auto model = project.Resources().Load<CRobotModelResource>("D:/assets/robot.fbx");
```

ResourceLoader 的注册动作由产品模块提供。ProductRuntime 会回放到产品级 ResourceLoaderRegistry；创建 Project 时再回放到项目级 ResourceLoaderRegistry。资源实例按 Project 隔离。

### 8.7 定义 Command

Command 是前端操作进入 backend 用例的入口。它位于 Mailbox 之上。

当前内置命令分为三层：

- `App.*`：应用级命令，发往 application mailbox。
- `Product.*`：产品级命令，发往 product mailbox。
- 产品自定义项目命令：发往 project mailbox。

产品自定义命令建议按产品协议定义主/子命令，并在产品命令模块中通过 `ICAX_REGISTER_COMMAND_TARGET` 注册 command target。ProductRuntime 加载产品 DLL 后会按模块路径回放命令目标注册。

```cpp
class CRobotCommandTarget final : public iCAX::Command::CCommandTarget
{
public:
    CRobotCommandTarget()
        : CCommandTarget("Robot")
    {
        Bind("Solve", [this](
            const iCAX::Command::CCommandRequest& request,
            iCAX::Application::IApplicationContext& app,
            iCAX::Product::IProductContext* product,
            iCAX::Project::IProjectContext* project) {
            return Solve(request, app, product, project);
        });
    }

private:
    iCAX::Command::CCommandResponse Solve(
        const iCAX::Command::CCommandRequest& request,
        iCAX::Application::IApplicationContext& app,
        iCAX::Product::IProductContext* product,
        iCAX::Project::IProjectContext* project)
    {
        (void)request;
        (void)app;
        (void)project;

        if (!product)
        {
            throw std::invalid_argument("Robot command requires ProductContext");
        }
        auto solver = product->GetServiceProvider().Resolve<IRobotSolverService>();
        solver->Solve();

        iCAX::Command::CCommandResponse response;
        response.nStatus = iCAX::Command::ECommandStatusCode::Ok;
        return response;
    }
};

ICAX_REGISTER_COMMAND_TARGET(CRobotCommandTarget)
```

ProductRuntime 会把 ApplicationContext、ProductContext、ProjectContext 直接传入命令函数。服务、Repository、资源库和 PDO 等能力从对应 context 获取，不再额外注入泛型命令上下文。

### 8.8 定义前端入口

产品前端入口由 manifest 的 `webpage.entry` 表达，并同步进入 `ProductDefinition.FrontendEntry`。`WebPageShell` 会动态加载入口模块，产品模块可以贡献产品页和项目页。

建议入口形态：

```js
export function mountProduct(context) {
  context.mount.innerHTML = "<div>Robot product workspace</div>";
}

export function mountProject(context) {
  context.mount.innerHTML = "<div>Robot project workspace</div>";
}
```

前端命令应通过语义 client 调用，不直接拼底层 mail：

```js
await appClient.startProduct("robot");
await appClient.openProjectFile("D:/projects/cell.robot");
await projectClient.execute(RobotCommands.solve, { mode: "fast" });
```

### 8.9 定义产品协议

每个产品应在 `protocol/` 下维护自己的协议说明。

建议至少包含：

```text
protocol/
  mailbox/
    README.md              命令名、type code、payload、response
  pdo/
    README.md              PDO id、数据结构、刷新频率、丢弃策略
```

协议文档要明确：

- 命令发往 application、product 还是 project mailbox。
- 请求 payload 字段。
- 响应 payload 字段。
- 错误码和错误语义。
- 是否可撤销、是否会写事务日志。
- 是否涉及资源 id/source。

## 9. 产品接入检查清单

新增产品时，至少检查：

- `ProductID` 唯一。
- `ProjectFile.Magic` 唯一且非空。
- 项目文件写入 magic，ApplicationHost 可快速探测。
- `webpage.entry` 指向产品前端入口，并能被 WebPageShell 动态加载。
- backend 模块路径正确，DLL 能被 `LoadLibraryA` 加载。
- Component 使用正确字段宏，并注册 creator。
- Behaviour 通过 `AUTO_REGIST_BEHAVIOUR` 注册。
- Service 通过 `AUTO_REGIST_SERVICE` 注册。
- ResourceLoader 通过 `ICAX_REGISTER_RESOURCE_LOADER` 注册。
- 自定义 Command 通过 `ICAX_REGISTER_COMMAND` 注册。
- 前端命令通过 Promise 等待 response mail。
- 高频状态使用 PDO，不用 Mailbox 刷屏。
- 大资源使用 Resource，不放进 Mail payload。
- 产品最近项目、用户偏好等运行数据放 ProductData，不放 ProductDefinition。
- 项目数据放 Repository，项目资源放 ResourceLibrary，应用配置放 ApplicationContext。

## 10. 当前实现边界

当前已经明确的边界：

- 单进程内运行。
- C++ backend 与 H5 frontend 分属不同线程。
- ApplicationHost 有应用级工作线程。
- ProductRuntime 不单独创建线程。
- 每个 Project 有自己的工作线程。
- Project 是数据隔离边界。
- Service 当前 Application 级共享；ComponentMeta、Behaviour 定义、CommandHandler 和产品级 ResourceLoader 按 ProductRuntime 隔离。
- Repository、ResourceLibrary、Universe 实例、project mail channel 按 Project 隔离。

当前仍建议继续完善的点：

- 真实宿主 bridge，把 H5 `IcaxBridge` 接到 C++ `CMailPostOffice`。
- PDO 前端订阅实现。
- 资源 URL 暴露机制，让 H5 可以预览模型、图片、文本等资源。
